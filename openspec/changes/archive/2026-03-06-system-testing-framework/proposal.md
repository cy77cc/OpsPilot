# Proposal: 系统测试体系建设

## Summary

为 k8s-manage 平台建立完整的测试体系，重点覆盖安全关键模块和核心业务模块，将测试覆盖率从 22% 提升至 50% 以上。

## Motivation

### 当前问题

1. **测试覆盖率低**: 总体覆盖率仅 22%，存在大量未测试代码
2. **安全风险**: 用户认证模块 (user/auth) 覆盖率 0%，存在安全隐患
3. **权限漏洞**: RBAC 模块覆盖率 14.5%，权限校验测试不完整
4. **E2E 测试不可用**: 需要认证环境，大部分测试被跳过
5. **测试不均衡**: 21 个模块覆盖率低于 30%

### 风险矩阵

| 模块 | 覆盖率 | 风险等级 | 影响 |
|------|--------|----------|------|
| user/auth | 0% | 极高 | 安全漏洞、账户被盗 |
| rbac | 14.5% | 高 | 权限泄露、越权操作 |
| cluster | 12.8% | 高 | 集群管理故障 |
| deployment | 18.5% | 较高 | 部署失败、服务中断 |
| node | 0% | 高 | 节点管理异常 |

## Scope

### In Scope

- 用户认证模块单元测试 (Login, Register, Refresh, Logout)
- RBAC 权限控制集成测试
- 集群管理模块测试
- 部署服务模块测试
- API 契约测试基础设施
- E2E 测试环境支持

### Out of Scope

- 性能测试 (k6) - 已有基础，待后续集成
- 前端测试补充 - 前端测试覆盖率较好 (64 个用例)
- CI/CD 流水线改造 - 单独变更处理

## Goals

1. **覆盖率目标**: 从 22% 提升至 50%
2. **安全模块**: user/auth 覆盖率达到 80%+
3. **权限模块**: rbac 覆盖率达到 70%+
4. **核心模块**: cluster, deployment 覆盖率达到 50%+
5. **E2E 可用**: 解决认证问题，E2E 测试可运行

## Approach

采用分层测试策略，按优先级分阶段实施：

```
Phase 1: 安全关键模块 (P0)
├── user/auth 单元测试
└── rbac 集成测试

Phase 2: 核心业务模块 (P1)
├── cluster 测试
└── deployment 测试

Phase 3: API 契约测试 (P2)
└── 接口稳定性验证

Phase 4: E2E 完善 (P2)
└── 认证环境 + 核心流程
```

## Success Criteria

- [ ] `make test-coverage-check` 通过 (覆盖率 >= 40%)
- [ ] 用户认证测试覆盖: 登录成功/失败、注册、Token刷新、密码验证
- [ ] RBAC 测试覆盖: 角色权限组合、超级管理员绕过、权限拒绝
- [ ] 集群管理测试覆盖: 导入流程、状态同步
- [ ] E2E 测试可运行: 无 skip 的测试用例

## Timeline

| Phase | 内容 | 预估工时 |
|-------|------|----------|
| Phase 1 | 安全模块测试 | 2-3 天 |
| Phase 2 | 核心模块测试 | 2-3 天 |
| Phase 3 | API 契约测试 | 1 天 |
| Phase 4 | E2E 完善 | 1 天 |

**总预估**: 6-8 天

## Risks & Mitigations

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 测试依赖外部服务 | 阻塞 | 使用 Mock，完善 testutil |
| 测试数据管理复杂 | 维护成本高 | 建立数据工厂模式 |
| E2E 环境不稳定 | 误报 | 增加重试机制和等待策略 |
