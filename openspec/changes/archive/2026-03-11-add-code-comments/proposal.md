# Proposal: 添加代码注释

## 问题

当前代码库缺乏注释，导致：
- 代码可读性差，新成员上手困难
- 复杂逻辑的意图不清晰
- 模块职责边界模糊
- 维护时需要反复理解代码

## 现状分析

| 模块 | 代码量 | 文件数 | 当前注释率 |
|------|--------|--------|------------|
| internal/service/ | 24,655 行 | 120 文件 | < 1% |
| internal/ai/ | 11,371 行 | 51 文件 | < 1% |
| internal/model/ | 1,687 行 | 26 文件 | < 1% |
| 其他模块 | 7,375 行 | 60 文件 | < 1% |
| **总计** | **45,088 行** | **257 文件** | **< 1%** |

## 解决方案

为所有 Go 文件添加四级中文注释：

1. **文件级注释** - 描述包职责、主要入口
2. **结构体/类型注释** - 说明用途和字段含义
3. **方法/函数注释** - 参数、返回值、副作用
4. **行内注释** - 解释复杂逻辑的"为什么"

## 范围

- 所有 `internal/` 目录下的 `.go` 文件（不含 `_test.go`）
- 约 257 个文件，45,088 行代码

## 文件类型分类

| 类型 | 示例文件 | 注释重点 |
|------|----------|----------|
| 路由文件 | routes.go | 端点分组、HTTP 方法 |
| Handler 文件 | host_query.go | 请求处理、响应格式 |
| Model 文件 | ai_chat.go | 字段含义、表关联 |
| DAO 文件 | user.go | 缓存策略、查询逻辑 |
| 中间件文件 | jwt.go | 执行流程、拦截点 |
| 工具函数 | jwt.go | 输入输出、错误情况 |
| AI 核心模块 | orchestrator.go | 架构图、执行流程 |

## 执行策略

分批执行，优先级从高到低：

1. **P1 - AI 模块** (internal/ai/) - 最复杂，刚完成重构
2. **P2 - Service 层** (internal/service/) - 业务核心
3. **P3 - 基础设施** (middleware, utils, dao, config)
4. **P4 - Model 层** (internal/model/) - 相对简单

## 验收标准

- [ ] 所有文件有文件级注释
- [ ] 所有公开类型有注释
- [ ] 所有公开方法有注释
- [ ] 复杂逻辑有行内注释
- [ ] 注释风格统一（中文）
